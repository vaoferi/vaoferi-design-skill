# Good Answer

Prompt:
"Ось реальний скрін форми редагування кампанії. Зроби так, щоб інтерфейс перестав розвалюватися."

Good response:

1. Спочатку знайду, який компонент, селектор і токен керують кожним полем на цій сторінці.
2. Перевірю існуючий CSS, перш ніж писати новий.
3. Якщо проблема вирішується через наявний стиль, зміню лише його або додам мінімальний варіант.
4. Використаю flat, traceable class naming, щоб ownership стилів був очевидний.
5. `!important` не використаю.
6. Якщо джерело стилю неочевидне, спершу трасую його до правила, а не приклею новий клас поверх.
7. Якщо компоненту треба адаптуватися в іншому контейнері, використаю `@container`, а не хаки на весь viewport.
8. Перевірю результат на desktop, tablet і mobile та покажу visual QA.

Expected behavior:

- root cause first;
- reuse existing CSS;
- keep class ownership obvious;
- no new visual language;
- no one-off hacks.
